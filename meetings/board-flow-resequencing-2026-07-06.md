## 1. 실제 업무 흐름 (Jin 확인)

1. 바이어가 채팅에 링크 던지며 문의 (살지 안 살지 미정) → 여러 링크 주고받으며 차 얘기
2. 차 선택 → **board "매입예정 추가" (이 시점 영업이 가진 건 링크뿐)**
3. 검차 (현지확인) → 사진/영상 확보
4. 영업이 바이어에게 **견적서** 전달 (그 나라 통화·할인 반영)
5. 바이어 수락/거절 → **수락해야만** 실제 매입
6. 국내 딜러에게 계약금 → 잔금 완납 → 차량 인수 (+ 딜러가 **사진 5장**[명함·차량정보·서류류] 넘겨줌)
7. won → car-erp 재고 전환 (연동 B)

핵심 원칙:
- **매입은 바이어 OK 후에만** (선매입 없음).
- **사진 5장 = 검차 사진과 별개** — 딜러에게 차를 산 뒤 받는 이미지(명함·차량정보 등). won 시점엔 무조건 손에 있음.

## 2. 금액 모델 (확정)

### 2.1 값의 정체
| 값 | 의미 | 시점/출처 | 가변? |
|---|---|---|---|
| **차값 원가** (`car_cost`) | 국내 매입값 = ERP 매입가 | **링크에서 자동추출된 금액을 고정값으로** (add 시 auto-fill) | 고정. 미세조정은 ERP에서 |
| 관례 할인율 (`discount_rate`, %) | 영업이 기본 제공하는 관례 할인 | 견적(전달) 화면 | sell-side only |
| **차감액** (신규, 절대금액) | 바이어가 추가로 깎아달라 할 때 | 견적(전달) 화면, 있을 때만 | sell-side only |

| 배송/운임 (`shipping_usd`) | 1640/1740/1840 USD 택1 | 견적(전달) 화면 | — |

### 2.2 계산식 (숫자 예시 · Model A)
```
차값 원가 (고정)              10,000,000원   ← ERP '매입탭' 매입값(원가 그대로)
− 관례 할인율(%)  예:5%        −  500,000원   ← sell-side (관례)
− 바이어 추가 차감액           −  300,000원   ← sell-side (신규, 있을 때만)
──────────────────────────────────────────
판매가(차값)                    9,200,000원   ← ERP '판매탭' 판매가 (매도비 미포함, 환율 적용)
+ 운임비 (배송 USD→통화)         ...           ← ERP '판매탭' 별도, ERP서 합산
──────────────────────────────────────────
총판매액 = 판매가 + 운임비 (ERP에서 합산)

[별도] 매도비 43/44만            ← ERP '매입탭' 비용. 바이어 미청구. 판매마진서 차감(회사 부담).
```
- **매입가(원가) > 판매가**여도 정상 → 차액은 **car-erp 9% 부가세(원가×9%)**가 수익으로 전환.
- 매입값 = **할인 이전 원가** / 판매가 = **할인 후 차값(매도비 제외)** / 매도비 = **매입탭 비용(회사 부담)**.

### 2.3 car-erp 전송 구조 (필드 그대로, 담는 값만 변경 — Model A)
- **매입탭**: 매입값(차값 원가) + 매도비 — 2개 (KRW)
- **판매탭**: 운임비 + 판매가(**할인 후 차값, 매도비 제외**, 환율 적용) — ERP가 합산해 총판매액
- 페이로드 필드(`purchase_price_krw` / `selling_fee_krw` / `sale_price` / `transport_fee`)는 **그대로**.

## 부록 A — ERP 마진 공식 + Excel 원본 대조 (2026-07-06 실측)

권위: `car-erp/app/Models/Settlement.php` + 설계원본 `바탕화면/0. 헤이맨 수출차량현황표.xlsx`(시트 `수출차량매입-2026`, 3349행). Excel 실데이터 row 3 수식:

| Excel | 항목 | 수식 | row3 값 |
|---|---|---|---|
| P | 구입금액(=매입값) | 입력(실제 원가) | 15,160,000 |
| Q | 매도비 | 입력 | 440,000 |
| R=BX | 매입금액(합계) | `P+Q` | 15,600,000 |
| AF | 판매금액(외화) | 입력 | 8,796 EUR |
| AK | 운임비 | 입력(판매통화) | 1,504 EUR |
| AD | 판매금원화 | `(AF+AH+AI−AJ+AK)×AG` | 17,446,240 |
| BY | 판매금원화(운임제외) | `AD−(AK×AG)` | 14,913,504 |
| CA | 정산판매금(비용공제) | `BY−BV(기타비용)` | 14,831,854 |
| **CB** | **판매마진** | `CA−BX` | **−768,146 (음수!)** |
| **CC** | **부가세마진** | **`P×0.09`** | **1,364,400** |
| **CD** | **총마진** | `(CB+CC)×0.9` | **536,628** |
| CF | 지급액 | 총마진 tier(음수=0/<100만=10만/…) | 100,000 |

**핵심 확증 (Jin 모델 = Excel 설계 = 정확):**
1. **판매마진이 음수**(−768,146) = 원가 이하 판매가 실제로 일어남.
2. **부가세마진 = 구입금액 × 9%**(1,364,400)가 손실을 덮어 총마진 +536,628 = 회사+영업 수익. ← "9% 부가세로 수익"의 실체.
3. **구입금액(P) = 실제 원가(할인 안 뺀 값).** 부가세마진이 P×9%라 P가 원가여야 환급이 정확.

**⇒ 현재 board 가 Excel 설계와 어긋나는 지점 (정확히):**

**(1) 매입값이 원가가 아님 → 부가세마진 과소.**
board 는 `purchase_price_krw = 차값−할인` 을 보냄 → ERP 구입금액(P)이 원가보다 작음 → 부가세마진(P×9%) 과소. Excel 은 P=원가.

**(2) 판매가를 원가에서 도출 → 판매마진이 사실상 무의미해짐 (핵심).**
board 는 `sale_price = 차값−할인+매도비` 로 **판매가를 매입원가에서 파생**시킴. car-erp 판매마진 = `정산판매금 − (구입금액+매도비)` 인데, 여기에 대입하면 **할인·매도비가 매입·판매 양쪽에서 상쇄되어 판매마진이 ≈ −기타비용으로 붕괴** — 실제 딜의 마크다운(원가 대비 얼마 깎였나)이 ERP 에 전혀 안 실림. 지금 ERP 에 도달하는 유일한 실질 마진 신호 = 부가세마진뿐이고 그마저 (1)로 과소.
→ **Jin 재설계의 진짜 의의**: 원가는 매입쪽 고정, 할인+차감액은 sell-side 로 **분리**해야 판매마진이 진짜 −768k 형태로 살아나고, 그걸 부가세마진이 메우는 Excel 경제학이 ERP 에 그대로 재현됨.

**(3) 매도비 — 마진 이중계상 아님, 그러나 판매합계는 부풀림.**
매도비는 위 상쇄로 **판매마진에선 상쇄돼 이중계상 아님**(오해 주의). 문제는 다른 데 있음: `sale_price` 에 매도비가 들어가 **`sale_total_amount`(=미수율 분모 + 2026-07-06 재설계의 FX baseline)이 부풀려짐** = 표시·미수율·환차 왜곡. Excel 은 매도비를 바이어 판매금액(AF)에 안 넣음(매입쪽 비용으로만). Jin 이 Q2 에서 "매도비가 총판매가에 포함 안 되지 않아?" 한 직감이 Excel 상 맞음.

**⚠️ 심각도 완화 프레이밍 (오해 방지):**
ERP 의 `purchase_price`/`sale_price`/`exchange_rate` 는 **sync 후 관리가 편집 가능한 pre-fill**(수신 컨트롤러 주석). 즉 board 역할 = **정확한 pre-fill 제공**. Jin 재설계는 "board 가 원장을 오염시킨다"가 아니라 **관리의 수기교정 부담·오류 위험을 줄이고 Excel 설계와 정렬**하는 것. 순 총마진 방향은 (2)의 붕괴 때문에 단순치 않으니 **개별 메커니즘(부가세마진 과소 + 판매마진 붕괴)만 단언**하고 순증감은 매도비 정리 후 확정.
