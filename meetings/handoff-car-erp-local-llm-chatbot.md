## 0. 한 줄 요약
Notion "사내 업무 가이드"를 근거로 답하는 **완전 로컬**(데이터 외부 전송 0) 챗봇을 만들었다.
현재 **A단계(업무 Q&A = RAG) 실동작 완료**. **B단계(super/admin이 매출·미수금 물으면 답 = car-erp DB 조회)는 미착수** — 이게 car-erp 몫.

## 1. 배경 & 확정된 결정 (Jin)
- **로컬 필수** — 민감정보. Claude/GPT API 배제(프라이버시 이유, 비용은 부차).
- **서버(Lightsail)엔 LLM 안 올림** — GPU 없고 라이브앱 상주. LLM은 **GPU 머신이 호스팅 + 앱이 API 호출** 구조.
- **범위 순서: A(업무가이드 Q&A) 우선 → B(매출·미수금 조회) 추후.**
- **상시 호스트 = Windows GPU 데스크톱** 확정. **최소 VRAM 8GB**(qwen3:8b 구동). RTX 2070 8GB로 검증 완료. 1060 3GB는 부적합(8B 안 올라감).
- Notion 지식 갱신은 **스케줄 sync**(예: 매일 밤 작업 스케줄러)로 자동화 예정. 실시간 아님.

- `think:false` + `<think>` 제거. 환각 방지 프롬프트(자료에 없으면 "없습니다").

## 5. ⭐ car-erp가 이어받을 것 — B단계 (매출·미수금 조회)
이게 car-erp 세션의 실제 작업. A와 **성격이 완전히 다름**:
- A = 문서 검색(RAG). B = **DB에서 실제 숫자 조회(Text-to-SQL / function-calling).** 숫자 틀리면 경영판단 오류 → 정확도가 생명.
- **매출·미수금 데이터는 car-erp 원장 소유.** (board 포털 미수금 탭도 car-erp 값 소비.) 그래서 B는 car-erp가 설계·소유해야 함.

권장 설계 원칙(board 세션 의견, car-erp가 판단):
1. **LLM이 직접 SQL을 짜게 하지 말 것.** 대신 `이번달_매출()`, `미수금_현황(바이어?)` 같은 **미리 검증된 조회 함수 몇 개**를 두고 LLM은 "어떤 함수 부를지"만 고르게(function calling). → 틀린 숫자·위험 쿼리 원천 차단.
2. **권한 게이트**: super/관리자만. 챗봇도 로그인 세션 권한을 태워 영업이 전체 미수 못 보게(car-erp의 본인격리·SoD 원칙 그대로).
3. **감사로그**: 누가 무엇을 물었는지 기록.
4. A와 B는 **분리된 파이프라인**. 같은 위젯에서 라우팅만.
