# 검차 영상 업로드 전략 (다음 세션 진입점)

> 2026-06-25 Jin과 논의. **설계만, 빌드 안 함.** Jin이 회의에서 "영상 업로드 필요 여부 + 촬영방식"을 정해오면 이 문서대로 진행. 운영 한도 상향(100MB)은 **이미 적용 완료**(아래 §3).

## 1. 배경 / 사용 맥락 (Jin 제공)
- 검차직원이 **현장에서 직접** 영상 업로드 → 바이어가 요청하면 전송(전달대기/회신 드로어의 사진공유 경로 = `PhotoController` 프록시 + `navigator.share`).
- 망: **LTE/5G, WiFi 없음** → **업로드 시간이 핵심 고통**.
- 직원 폰: **아이폰·갤럭시 혼재, 사파리·크롬 혼재.** 외국인 직원 → "매 촬영 설정 바꿔라"는 비현실적.
- 영상 크기: 갤럭시S25 기준 **1:50 = 174MB**(≈1.5MB/s, ~12.6Mbps, 4K/고비트레이트). 길이 대부분 **30초~1분**, 가끔 몇 분, 2분 잘 안 넘김. 차 1대당 **5~6개**.

## 2. 핵심 트레이드오프 (꼭 이해하고 결정)
- **서버 압축(FFmpeg)은 "업로드 시간"을 못 줄인다** — 원본이 일단 다 올라가야 압축. 저장용량·바이어전송 크기만 줄임.
- **업로드를 빠르게 = 보내는 바이트를 on-device로 줄이는 것뿐.**
- `ffmpeg.wasm`(브라우저 후처리 압축)은 모바일선 비현실(느림·OOM·iOS 취약).

## 3. ✅ 이미 적용 — board 업로드 한도 100MB (2026-06-25)
원래 서버 PHP-FPM이 40M라 30초~2분 영상·다중사진이 막혔음. board만 상향:
- `/var/www/board/public/.user.ini`: `upload_max_filesize=100M` + `post_max_size=110M` (perdir → **car-erp 무영향**, php.ini 공유라 글로벌 안 건드림).
- board nginx `client_max_body_size 100M→120M`.
- **웹 실효값 확인됨**(https): 100M. 배포(`git reset --hard`, clean 없음)가 `.user.ini` 보존. 상세=`deploy-runbook.md §B-3b`.
- → **개별 100MB·요청합 110MB까지 OK** (30초~1분 영상·다중사진 통과). **2분/174MB는 여전히 초과** → 아래 on-device 압축 필요.
- ⚠️ 두 한도 구분: `upload_max_filesize`=파일1개당 / `post_max_size`=한 요청 전체. **Livewire는 다중선택 파일을 한 요청에 묶어 전송** → 합이 post 넘으면 통째 실패.

## 4. on-device 압축 3안 (회의 결정 → 택1/조합)
- **(C) 폰 촬영 4K→1080p 설정**: 174MB→~30-40MB(1/4~1/5). 무료·즉효, **브라우저 무관**(폰 설정이라 사파리도 OK). 매 촬영은 비현실 → **온보딩 때 1회 설정**이 현실적. Jin이 "쉬워진다"고 본 안.
- **(A) board 앱 내 녹화 버튼**(getUserMedia+MediaRecorder를 720/1080p·저비트레이트 강제): 직원 행동 0, 처음부터 작게. **크롬(안드)은 잘 됨, iOS 사파리는 MediaRecorder 코덱 제약**이 변수. → Jin: "크롬 쓰면 board 촬영 제공" 가능.
- **(B) 기본카메라 촬영 후 WebCodecs로 업로드 직전 자동압축**: 촬영UX 익숙하나 기기/브라우저 편차 큼·iOS 위험·구현복잡. 비추 우선순위.
- 현실 조합 후보: **사파리/아이폰 = (C) 1080p 설정 / 크롬·갤럭시 = (A) 인앱 촬영.** + 서버 FFmpeg(저장·전송 경량화 보완재, 큐워커 이미 가동).
