## 1. 한 줄 요약

바이어는 **ssancar.com**(여러 사이트 매물을 API로 모은 카탈로그)에서 차를 보고 채팅을 건다 → **respond.io**가 받아 응대 →
"살게요" 하면 SSANCAR이 그 차를 **그때 매입**(경매/구매 = purchase-board) → 사고 나면 **car-erp**에 등록되어 정산·통관까지 흐른다.
이 네 시스템을 **매물번호(c_no) 하나로 꿰면** 처음 문의부터 거래완료까지 자동으로 이어진다.
검토 결과 **거의 전부 지금 요금제(Growth)로, 추가 돈 없이 가능**.

## 2. 시스템 4개 (쉽게)

| 시스템 | 역할 | 상태 |
|---|---|---|
| **ssancar.com** | 여러 중고차 사이트 매물을 **API로 모아 보여주는 카탈로그**. 바이어가 여기서 차 봄 | 작동 중 (별개 사이트) |
| **respond.io** | 바이어가 WhatsApp/FB/IG로 채팅하면 받아주는 메시징 도구 | 작동 중 (외부 서비스) |
| **purchase-board** | "살게요" 한 차를 실제로 매입(검차·경매)하는 업무보드 | 아직 껍데기(목업) |
| **car-erp** | **산 뒤부터** 차량 재고·정산·통관·B/L·DHL 관리하는 본체 원장 | 작동 중 (우리가 만든 것) |

**중요한 사업 구조:** ssancar.com 매물은 SSANCAR 소유가 아니다. **여러 사이트에서 긁어모은 카탈로그**다.
→ SSANCAR은 재고를 쌓아두는 게 아니라, **바이어가 원하면 그때 사 오는 "수주 후 매입" 구조**.
→ 그래서 car-erp는 "산 차"만 있고, 바이어가 보는 매물 전체는 없다. (AI가 car-erp 재고를 알 필요 없는 이유)

## 3. 모든 걸 꿰는 실 = 매물번호 (c_no)  🧵

바이어가 ssancar.com에서 차 보고 채팅 누르면 — **그 차 정보가 자동으로 따라온다** (2026-06-04 실제 확인).

실제 들어온 채팅 예시:
```
GENESIS GV70 2.5T AWD Basic · 2022.03 · 41,189km · Gasoline
Price: 26,005 USD
Hi SSANCAR, Is this car still available?
GENESIS GV70 2.5T AWD Basic (#6797296)   ← 매물번호 c_no
41,189 km · Gasoline · Plate: 191누7710   ← 번호판도 옴
사진 첨부됨 · Ref: SSC-WEB · Source: chatgpt.com
```

이 **`6797296`(c_no)** 이 처음 문의부터 끝까지 따라가는 식별번호.

```
ssancar.com 매물 (c_no 6797296)
   │ 바이어 채팅 → 이 번호+사진 자동으로 옴
   ▼
respond.io (이 사람 = 6797296 차 관심)   ← AI가 바로 어느 차인지 앎
   │ "살게요" → SSANCAR이 그 차 매입
   ▼
purchase-board (c_no 6797296 = VIN 매칭)
   │ 낙찰/구매
   ▼
car-erp (이 차 + 이 바이어 연결됨, c_no도 기록)
   │ 입금되면
   ▼
respond.io 그 바이어 단계 자동전진 (연동 C)  ← 누구 단계 올릴지 안 헷갈림
```

**효과:** 회의 때 엔지니어가 걱정한 "문의한 바이어 = 실제 산 사람이 같은지 보장할 키 없음" 문제가
**c_no가 처음부터 따라오니 자동 해결.** → SSANCAR이 차 살 때 **c_no를 car-erp에도 기록**해두면 한 줄로 이어짐.

## 4. 연결선 A · B · C

```
ssancar.com ─ respond.io ─── A ─── purchase-board ─── B ─── car-erp
                  └─────────────── C ──────────────────────────┘
```

| 연결 | 무엇 → 무엇 | 쉽게 | 지금 가능? |
|---|---|---|---|
| **C** | car-erp 입금/판매 → respond.io | 돈 들어오면 바이어 단계 자동 Payment·Customer | 🟢 **지금 (최우선)** |
| **B** | purchase-board 낙찰 → car-erp | 산 차가 car-erp 자동등록 (= 지금 카톡 수동작업 대체) | 🟡 앱 만든 후 |
| **A** | purchase-board 검차 ↔ respond.io | 실차 검수사진을 바이어에게 보내고 답 받기 | 🟡 앱 만든 후 |

**현재 수작업(연동 B가 없앨 것):**
지금은 경매/구매되면 → **[관리]에게 카톡으로 차량정보 전달 → [관리]가 car-erp에 수동 등록 → 워크플로우 시작.**
연동 B = 이 카톡·수동입력을 자동화.

## 5. AI 레이어 (Jin의 "완전 자동화" 비전)

respond.io AI 에이전트(SSANCAR Front Desk)에 과거 대화·문서 학습 →
- 🤖 밤에도 1차 응대. **바이어가 어느 차 문의했는지 c_no로 이미 알고 시작.**
- 🤖 돈·할인·판단 나오면 → 스스로 사람(영업) 호출 (handoff)
- 🤖 산 뒤 "내 차 어디?" → car-erp 조회해 정확히 답

**AI가 아는 법 (중요):**
- 잘 안 바뀌는 것(회사소개·수출절차·보증·FAQ) → 📚 **학습**
- 계속 바뀌는 것(배송상태·정산) → 📖 **car-erp 실시간 조회**
- "이 차 아직 있어요?"(첫 질문) → car-erp 아니라 **ssancar.com(원본 사이트) 재고** 확인 사항
- **재고 전체를 AI가 외울 필요 없음** — 바이어가 c_no로 콕 집어 오니까

**완전 자동화 = 비율 문제.** AI ~75% / 사람 ~25%. 돈·낙찰·입금확인은 항상 사람.

**AI 켜기 전 안전장치 3개:** ①약속 제한(멋대로 할인 금지) ②개인정보(과거대화 타 바이어 정보) ③실시간 데이터 통로(car-erp 읽기).

## 7. 절대 넘으면 안 되는 선 (개인정보·법)

respond.io = 해외 클라우드 → 개인정보 보내면 국외이전(개인정보보호법 §28).
**respond.io에 보내면 안 됨:** 판매자 주민번호·이름·주소·계좌 / 서류 사진(등록증·성능지·말소신청서) / (우리가 보내는) 번호판·금액.
**해결:** 사진은 차량 외관만 · 보내는 정보는 (바이어ID·단계·차량ID)만 · respond.io와 위탁계약(DPA).
※ 채팅에 **들어오는** 번호판(매물에 원래 공개된 것)은 우리가 보내는 게 아니라 받는 것이라 성격이 다름.
